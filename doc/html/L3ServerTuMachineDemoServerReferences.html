References

Until now we looked on the object nodes in a very hierarchical view. Jobs can be added to and removed from the production plan, and other collections or objects with dynamic children could be added. But sometimes it is useful to link two existing nodes by reference. Those nodes do not need to be in a hierarchical relation-ship. In this example, a reference targeting the currently active job is added directly to the glass tempering machine instance. Once the active job is finished the reference is removed again.

The property ActiveJobId contains the NodeId of the active job, or null if there is no job being processed.

[UaReference(ReferenceTypeId = ReferenceTypes.HasActiveJob, NamespaceUri = Namespaces.MachineDemoServer)]
public NodeId ActiveJobId
{
get => m_activeJobId;
private set => SetField(ref m_activeJobId, value);
}
private NodeId m_activeJobId;

The attribute UaReference signals the BindModel mechanism that a reference should be created or deleted, whenever the property changes. The ReferenceType can be set by ReferenceTypeId argument. Here a custom reference type is used that was defined in the NodeSet2.xml file before.

The value of the property is the target node id, the source node id is the node id of the containing object node or variable node. In this example, the node id is given as a NodeId, but ExpandedNodeId can be used equally. In fact, one can even use IEnumerable<NodeId> or IEnumerable<ExpandedNodeId> if an entire list of references of the same reference type is needed.

In order for a connected client to know that a reference has been added or removed, the general model change event must be raised and the node version changed, just as in the previous chapter. Again, the work is done by the BindModel mechanism as long as a NodeVersion node is present or the UaAutoNodeVersion attribute is set.
