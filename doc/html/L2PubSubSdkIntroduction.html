Introduction

Traditionally, the OPC UA communication consists of a server and a client. The client requests data to be delivered or other services from the server. The server sends responses to meet the requests. In many cases, this approach is exactly what is desired. But it comes with some downsides which the OPC UA Specification circumvents with OPC UA Part 14: PubSub. The client needs to know the server, how to access it1 and has to select the data of interest itself. Furthermore, if there are more than one client asking for the same or similar data, the server has to respond every client on its own. This can become time-consuming for a large number of clients.

The OPC UA PubSub communication consists of publishers, subscribers and the Message Oriented Middleware. This Message Oriented Middleware can be network infrastructure like switches if UDP multi-cast is used as transport protocol or could be a message broker if MQTT is used as transport protocol. For the sake of convenience, only the term broker is used in the following. The publisher sends data of interest to the broker and the subscriber receives the data from the broker. The publisher, acting as data source, has a similar role like a server. The subscriber, acting as consumer of the data, has a role comparable to the client. While the publisher and the subscriber know the broker, they do not necessarily know each other. A more in-depth introduction can be found in the chapter OPC UA PubSub Messaging Concept.

A possible use case is a generic data collector, that records the data from several applications/servers. The server publishes the data of interest, that was previous configured by a administrator that knows the server and its content very well, that might be the process engineer supervising the machine belonging to the server or someone else having the domain specific knowledge. The data collector, on the other hand, does not need to know the machine or its server. It only subscribes to the broker.

Of course the subscriber has not the full set of possibilities as an regular OPC UA client. Most service sets are not covered by pub-sub. Fortunately, OPC UA offers both PubSub and the Service Sets.

Features

The current PubSub implementation supports the following Features:

  • Mapping
    • JSON NetworkMessages/DataSetMessages
      • Encoding of reversible/non-reversible messages
      • Decoding of reversible messages
      • All NetworkMessageContentMask combinations (including SingleDataSetMessage, DataSetMessageHeader and NetworkMessageHeader)
      • All DataSetMessageContentMask combinations
      • All DataSetFieldContentMask combinations
    • MQTT
      • MQTT Version 3.1.1 and 5.0
      • TLS
      • Authentication via KeyCredentialManagement, i.e., ResourceUri and AuthenticationProfileUri
      • Connection properties of 1.05.02
      • Message properties of 1.05.02
      • Gzip compression
    • UADP
      • Fixed and dynamic layout
      • Encryption
  • Address space
    • PubSubConfiguration
    • Diagnostics
    • SecurityKeyServices

The libraries

The following section explains which functions the individual libraries perform in relation to PubSub.

The UnifiedAutomation.UaBase library provides the common pub-sub interfaces DataSetMessage, IPubSubConnectionProvider, IPubSubConnection and all other interfaces derived from IPubSubElement. Those are used by the UaServer library and implemented by the UaPubSub library. Furthermore, the UaBase library provides the PubSub related DataTypes.

An OPC UA server can act as a publisher and subscriber. The configuration interface is defined in Part 14 of the OPC UA specification. It is vendor independent and can be implemented by a generic client. The widely used UA Expert, for example, can be used. The UaServer library provides the PublishSubscribe Object in the server address space, including the configuration interface.

The UnifiedAutomation.UaPubSub library provides the implementation of the IPubSubConnectionProvider interface and the UADP message encoding. The encoding and transport-specific implementations are added via the AddProfile method. For example, the UnifiedAutomation.UaPubSub.Mqtt library offers the implementation for the JSON message encoding and the MQTT transport connection. Support for UDP will be added in later releases. Even though it is already possible to develop a custom transport connection, we do not recommend doing so at the moment as the interface is still in flux. In the case of custom message encodings, the fact that custom configuration data types are needed makes this even more difficult. These are neither supported by UaServer SDK nor by the configuration tools. We therefore strongly advise against this.

The UaClient library supports to download pub-sub configurations to a server via the PubSubConfigurationModel or PubSubConfigurationObjectStream classes, and the library supports to pull PubSub security keys from a SecurityKeyService via the RemoteSecurityGroupProviderService.
